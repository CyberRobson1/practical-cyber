

# 🛡️ Cybersecurity Research & Documentation Lab

This repository serves as a technical record of my practical application in cybersecurity. While I maintain a strong theoretical foundation in networking, protocols, and attack patterns, this space is dedicated exclusively to hands-on execution and evidence-based learning.

"Theory is the map, but the lab is the terrain." This is where I apply my knowledge to real-world problems, moving beyond tutorials to actual technical mastery.

---

## 📂 Laboratory Structure
To ensure auditability and clarity, every lab follows a standardized directory architecture.

```text
📂 [topic-name]/
├── README.md          # Technical summary, findings, and security impact
├── lab-notes.md       # Raw execution log (commands, errors, and observations)
├── 📂 screenshots/    # Annotated visual evidence of findings
└── 📂 evidence/       # Raw artifacts (pcap, logs, tool outputs)
```

### 📝 Documentation Standards
*   **README.md:** A high-level overview of the objective, the steps taken, and the technical conclusion. Screenshots are referenced and explained here, not just dropped in.
*   **lab-notes.md:** An honest, unfiltered record of the process. This includes failed commands, configuration hurdles, and the troubleshooting process. **It is messy by design to reflect the reality of the work.**
*   **Annotated Evidence:** No raw screenshots. Every image is marked up with highlights or captions to pinpoint the exact technical indicators or vulnerabilities discovered.

---

## 🧪 Methodology
Every lab entry is governed by a consistent six-step workflow to ensure total understanding:

1.  **Define the Question:** Establishing a single, clear technical goal before starting.
2.  **Active Execution:** Real-time logging of all commands and system responses in `lab-notes.md`.
3.  **Evidence Capture:** Saving raw files and descriptive screenshots (e.g., `unencrypted-creds.pcap`) immediately upon discovery.
4.  **Technical Annotation:** Marking evidence to demonstrate a logical understanding of the output.
5.  **Synthesis:** Compiling the final `README.md` once the process is understood and repeatable.
6.  **Intentional Commit:** Using descriptive commit messages that reflect the specific technical milestone achieved.
